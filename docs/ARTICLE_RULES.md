# SEO記事 単一正本ルール (ARTICLE_RULES.md)

このファイルは、SEO記事を作るときに **AI（Claude API・Claude Code・人間）すべてが従う唯一の正本**。
他のファイル（旧 `prompts/generate_article.md`、`PIPELINE.md` の HTML ルール、`.claude/projects/.../memory/feedback_*.md`）の内容は、すべてこのファイルに統合されている。**このファイル以外を参照しなくてよい**。

矛盾を見つけたら **このファイルを直す**。他のファイルにルールを追記しない。

---

## 0. 最上位原則 — このプロジェクトのゴール

**「指定キーワードで Google 検索1位を取れる記事」を作ること。**

そのための核心戦略は1つだけ:

> **1位記事の構成・内容を忠実にトレースし、表現だけ書き換える。**

- 1位記事 = 入力された競合スクレイピング `article_1_structure.md`（スコア最上位 = SERP上位）
- 「**構成・内容**」 = H2/H3 階層・順序、各セクションの文字数、ブロック種別（テーブル/リスト/段落/マップ/口コミ/CTA）、扱うトピック、院構成
- 「**表現だけ書き換える**」 = 見出し文言、本文表現、段落の言い回し、比較表のキャッチコピー、口コミの文章

このゴールに反する判断はすべて却下する。例えば「1位記事に存在しない H2 を AI が思いつきで追加する」「1位記事が単独LP（比較記事ではない）でも比較構造を勝手に作る」「文字数を AI が削る」は禁止。

## 1. 1位記事トレースの具体ルール（**最重要・最初に守る**）

### 1-1. 1位記事は機械的に決まる
- スクレイピング `summary.json` の `competitor_summary.top_article_profile` が1位記事のプロファイル。
- AI が「これは1位記事ぽくないから」と判断して別記事をトレース元に切り替えない。

### 1-2. 1位が単独LPか比較記事かを最初に判定する
1位記事の構造を見て以下のいずれか:

| 1位の型 | 判定基準 | 取り扱い |
|---|---|---|
| **比較記事型** | H2にクリニック比較系（「○選」「比較」「おすすめ」）があり、院別 H3 が3院以上ある | 通常トレース。H2/H3 を1位に揃える |
| **単独LP型** | 1院のページで、H2 は院プロフィール（基本情報・料金表・症例・特徴）中心 | **2位の構造をトレース元に切り替える**。1位の本文情報は「メイン院」として該当院 H3 に詰める |

判定は Step2 で行い、タグ構成設計書の冒頭に `**1位タイプ**: 比較記事型 / 単独LP型` を明記する。

### 1-3. H2 はトレース元の論点と順序を踏襲（数は集約OK）
- **H2 の順序と扱う論点** は、トレース元（通常は1位、単独LPなら2位）と一致させる。
- 各 H2 で扱う **論点** は同じ。見出しの文言は新表現に書き換える（§3-10 のコピーライティングルールに従う）。
- 競合 H2 採否マトリクスをタグ構成書冒頭に必ず出す（後述 §3-2）。

**H2 の数について（§1-9 で詳述する集約・追加の判断ルール）:**
- トレース元の H2 数と本記事の H2 数は **必ずしも一致させなくてよい**
- 集約・追加は §1-9 のルールに従って機械的に判断する

### 1-4. H3 数と順序もトレース元に揃える
- 各 H2 配下の **H3 数は ±1個まで**。順序は同じ。
- クリニック比較記事のクリニック個別 H3 は、トレース元の **掲載順をそのままトレース** する（推奨順を勝手に並べ替えない）。1位が「料金順」で並べていたら本記事も料金順。
- 院数（掲載クリニック数）はトレース元と一致 (±1院まで)。

### 1-5. 各セクションの文字数はトレース元の ±20%
- タグ構成設計書の各 H2/H3 に「**目標文字数**: ○○字（1位の○○字の±20%）」を機械的に書く。
- AI に「想定文字数: 22,000〜26,000字」のような幅を書かせない。**数字はトレース元から機械的に決める**。
- 全体の総文字数も 1位記事の `total_chars` の ±20%。

### 1-6. ブロック単位のトレース（最重要・H2より細かい粒度の必須ルール）

H2 だけをトレースしても、各 H2 内のテーブル・引用・画像・無見出しブロックが抜けると 1 位記事と情報量で勝負にならない。本記事はトレース元の **すべてのブロック** を順序通り・同じ構造で配置する。

#### 1-6-A. ブロック列マップを必ず作る（タグ構成設計の前提作業）

**ブロック列マップは必ず `scripts/extract_block_map.py` を使って機械抽出すること。AI の目視抽出はブロック種別 (H4 ラベル・装飾ボックス・段落数・テーブル列) を頻繁に取りこぼすため、機械抽出を必須化する。**

実行手順:

```bash
python3 scripts/extract_block_map.py \
  --url <1位記事のURL> \
  --out output/<keyword_slug>/_block_map_top1.md
```

出力された `_block_map_top1.md` をタグ構成設計書の冒頭からリンクし、各 H2/H3 の本文を書く前に必ず参照する。マップに記録されているブロックの種別・段落数・テーブル列数・項目数・ラベル見出しを、本記事でも同じ構造で配置する (§1-6-B〜E)。

**機械抽出マップに含まれる情報:**

- 各ブロックの種別 (P/H2/H3/H4/UL/OL/TBL/QUOTE/BOX/IMG/FIGURE)
- 段落の連続数 (P×N) と合計文字数
- 見出しのテキスト (どの位置のブロックか同定するため)
- テーブルのカラム名と行数
- リストの項目数
- 画像の alt またはファイル名

**段落の太字ラベル (例: 「○○のおすすめポイント」のような短い P) は構造上 H4 ではなく P×1 (20〜30字程度) として抽出されることがある。** 機械抽出マップで「**短い P (20〜30字) + 直後に長い P (500字級) または UL**」というパターンが続く場合、その短い P は本記事では小見出しラベルとして再現すること (H4 にするか、太字 P で再現するかは見た目を 1 位記事に揃える)。

**機械抽出を使わずに AI 目視で済ませた場合は §1-6 違反**。タグ構成書のレビュー時に `_block_map_top1.md` が存在することを必ず確認する。

(参考) 旧仕様で書き出していた粒度の詳細:

- リード文（**段落数を必ず数字で記録**: 「P×2」「P×3」のように）
- 段落（**連続する段落は個数を明示**: 「P×4」と書き、合計4個の段落があると分かるようにする。1個なら「P×1」と書く）
- テーブル（**カラム名と行数を記録**: 「TBL: カラム=[A/B/C]、N行」）
- 番号付きリスト・箇条書き（**項目数を記録**: 「UL×6項目」「OL×3項目」）
- 画像（カオスマップ・選定基準画像など本文流用しない画像も含む）
- 引用ブロック（**出典名と数を記録**: 「QUOTE×2(厚労省/日本皮膚科学会)」）
- ボックス（「この記事のまとめ」「結論ボックス」「コラム枠」など）
- 早見表・タブ式コンテンツ
- 目次（自動／手動）
- CTA ボタン
- H2 見出し（H2 自体もブロック列の1要素として記録）
- H3 見出し（同上）

**ブロック列マップは「タグ構成設計書の冒頭」に必ず出力する**（§3-1 の必須項目に追加）。これがないとブロック単位トレースが検証できない。

**重要**: 段落数・テーブル行数・リスト項目数は **必ず数字で**記録する。「P+QUOTE+P」のような曖昧な書き方は禁止 (段落が連続して何個あるかが不明なため)。**「P×3 → QUOTE → P×2」** のように書く。

**さらに重要**: リストやボックスの直前に **ラベル見出し (h4・小見出し・タイトル div) がある場合は必ず記録する**。1位記事ではしばしば `<h4>○○のおすすめポイント</h4>` のような小見出しがリストの前に付いている。これを見落として `UL×3項目` とだけ書くと、本記事ではラベルなしの裸の箇条書きになってしまう。

例 (正しい記録):
```
[H3] DMMオンラインクリニック
  - H4「DMMオンラインクリニックのおすすめポイント」
  - UL×3項目
  - P×4
  - TBL: 4カラム/8行
  - CTA
  - P×1 (注釈)
```

ラベル見出しを見落とすと、本記事では「**箇条書きの意図 (おすすめポイントを強調する装飾)**」が消えてしまい、視覚的なメリハリと読みやすさが大きく下がる。

#### 1-6-B. 「無見出しブロック」を勝手に H2/H3 に格上げしない

1 位記事には「H2 の直下、または H2 と H2 の間に置かれた**見出しなしのテーブル・ボックス・画像**」が頻出する。例:

- リード文の直後にある「対面 vs オンライン比較テーブル」
- 「料金比較表（予防/発毛タブ）」
- 「この記事のまとめ」ボックス（記事冒頭付近の早期結論）
- カオスマップ画像

これらは**そのまま無見出しブロックとして、1 位記事と同じ位置に配置**する。新規 H2 や H3 を起こして格上げしない。読者が「目次に出ないけど画面で見える」要素はそのまま再現する。

#### 1-6-C. テーブル内容のトレース

トレース元のテーブルがある場合、本記事も同じ構造で配置する:

- **カラム名は同じ意味**で配置（語彙は変えてよい）
- **行数は同じ**（ただし掲載クリニック数の調整は §1-4 に従う）
- **数値・料金・住所などの事実情報はそのまま使う**（§1-7 で許容）
- **キャッチコピー・修飾語は別表現に書き換える**

#### 1-6-D. ブロックの有無に関する個別ルール

以下は機械的に判定する:

| ブロック種別 | 採否判定 |
|---|---|
| 一覧ボックス（`clinic-index-box`） | 1 位の `intro_has_index_box` に揃える |
| 口コミブロック | 1 位の `h3_with_reviews_pct` ≥ 50% なら入れる、< 50% なら入れない |
| Google マップ | 1 位の `h3_with_map_pct` ≥ 50% なら入れる（オンライン専門院は除外） |
| 公式 CTA ボタン | 比較系では原則入れる（CV 直結） |
| カオスマップ画像 | 1 位にあれば配置（本サイトで未作成なら §1-9 B により不採用で OK） |
| 監修者プロフィールボックス | §1-9 B により本サイトで監修者がいなければ不採用 |
| 「この記事のまとめ」結論ボックス（冒頭の早期結論） | 1 位にあれば**必ず配置**（読者の離脱防止に直結） |

#### 1-6-E. ブロック追加・削除・順序変更の禁止

§1-9 で許容されているケース（クリニック個別 H2 の H3 集約・本サイト未実施コンテンツの不採用・FAQ/まとめ H2 追加）**以外で**、トレース元にあるブロックを削除したり、トレース元にないブロックを追加したり、ブロックの順序を変えたりすることは禁止。

迷ったら **1 位記事のままトレース**する。

### 1-7. 「表現は完全に書き換え」厳守
**禁止**:
- トレース元の見出し文言を丸ごとコピー
- トレース元の本文段落を「言い換えコピー」（語順だけ変えて意味を保ったもの）
- トレース元の固有のキャッチコピー・比喩・誇張表現の流用
- トレース元の口コミ文章をそのまま転載

**やる**:
- 同じトピックを別の語彙・別の語順・別の文体で書く
- 数値・料金・住所・診療時間は事実なのでそのまま引用OK（事実情報は著作物ではない）
- 1位記事のクリニック並び順とは違う順序にする（推奨ロジックの軸を別にする）

### 1-8. 「客観整理メタ表現」の禁止
本文に以下の表現を **使わない**:
- 「参考元では○○と整理されていました」
- 「比較しやすい特徴」「見られやすい声」「比較しやすい候補」
- 「○○として整理されています」
- 偽口コミラベル「参考整理」「比較整理」「総合相談で見られやすい声」など

なぜなら 1 位記事はこんなメタ表現を書いていない。**1位記事と同じトーンで断定的に書く**。

### 1-9. 構成判断のグレー領域（チャットごとのブレ防止）

トレース元の構造をどこまで忠実に踏襲し、どこから独自判断してよいかを **明示する**。これを書かないと AI（または別のチャットの同じ AI）がチャットごとに違う判断をして、記事構成がブレる。

#### A. クリニック個別 H2 の集約は **OK**
- 比較記事で 1 位記事が「クリニック × 院数分のH2」(例: 17クリニック = 17個のH2)で並べていても、本記事は **単一の比較H2 + H3 × 院数分** に集約してよい
- 理由: WordPress の自動目次が大量のH2で並ぶより、H3に集約した方が読みやすい
- ただし **掲載クリニックの数・順序は1位記事と一致** させる（§1-4 通り）

#### B. 「本サイトで実施していないコンテンツ」は **不採用** でOK
- 1位記事に「独自調査(85名アンケート)」「症例写真」「医師監修コメント」など本サイトで実体がないコンテンツがある場合、**不採用**でよい
- 虚偽コンテンツを生成する方がリスクが高い（E-E-A-T と法務観点）
- 採否マトリクスでは `❌ 本サイトで実施していないため不採用` と明記する
- 将来本サイトで実施した時に追加する想定で構造の余地は残してよい

#### C. SEO・UX 観点での H2 追加は **OK**
- 1位記事に **FAQ・まとめセクションが無い** 場合でも、本記事には追加してよい
- 理由: ユーザーニーズが明確に存在し、Google も FAQ schema/まとめを重視する傾向
- ただし **長くしすぎない**: FAQ は最大 Q&A 5個（§3-7 通り）、まとめは2段落程度

#### D. 上記以外の独自判断は **禁止**
- A〜C 以外で「読みやすさのため」「SEOのため」と独自に構造を変えるのは禁止
- 迷ったら **1位記事のままトレース** する。新ルールが必要だと感じたらこのファイル(§1-9)に追記する

## 2. 検索意図に応じた記事タイプ分岐

1位記事の型がいずれであっても、検索キーワードの **検索意図** から記事タイプを判定する。

| 検索意図 | キーワード例 | 記事タイプ | 主軸構造 |
|---|---|---|---|
| 比較・おすすめ系 | おすすめ / 比較 / ランキング / クリニック | クリニック比較記事 | 院別 H3 + 比較テーブル |
| Know系 | 副作用 / 効果 / 原因 / 仕組み / いつから / やめると | 解説記事 | 論点別 H3 |
| 費用・料金系 | 値段 / 相場 / 費用 / いくら | 費用解説記事 | 費用パターン別 H3 |
| 方法・対策系 | 治し方 / 予防 / 改善 / 対処法 | 対処法記事 | 手順別 H3 |

検索意図に合わない構造を1位記事から踏襲しない（例: Know系キーワードでも1位が比較記事なら、論点別 H3 に整理し直す）。

### 3-2. 競合H2採否マトリクス（必須）

タグ構成設計書の冒頭に、競合3記事（または取得した全競合）の**すべてのH2**を列挙したマトリクスを出す。

```markdown
## 競合H2採否マトリクス

| 競合H2（出典記事） | 採用 | 採用先 / 不採用理由 |
|---|---|---|
| 「○○の選び方」（1位記事） | ✅ | [H2] 同名トピックを別表現で採用 |
| 「○○とは」（2位記事） | ✅ | [H2] 「○○とは｜...」として採用 |
| 「○○の値段」（3位記事） | ❌ → H3で吸収 | [H2] 費用内訳 配下の [H3] 用量別の薬代相場 に吸収 |
| 「○○ランキング」（1位・2位共通） | ❌ | 検索意図と合致しないため不採用 |
```

**ルール:**
- 競合スクレイピングの全H2を漏れなく列挙
- 採用列は「✅」「❌ → H3で吸収」「❌」のいずれか
- 不採用とする場合、理由は以下のいずれかに限定:
  1. 他のH2/H3で吸収する（吸収先を具体的なH2/H3名で明記）
  2. 検索意図と合致しない（理由を1行で説明）
- 「なんとなく不要」「テンプレに無いから」は不採用理由として認めない
- マトリクスと本編H2が矛盾する場合はマトリクスを修正してから本編に進む

### 3-3. メインキーワードの H2/H3 含有

- **全H2のうち最低50%以上**にメインキーワード（または構成語「AGA」「治療」など）を含める
- **全H3のうち最低30%以上**にメインキーワード or 構成語を含める
- 同義語（例「AGA = 男性型脱毛症」「治療 = 施術」）も活用してバリエーション

### 3-4. titleタグの設計（コピーライティングルール）

titleタグは検索結果に表示される最重要要素で、CTR と SEO 評価を両方左右する。H2 のコピーライティングルール（§3-10）と同じ思想で組み立てる。

#### 文字数
- **1位記事のタイトル長の ±5 文字** を基本とする
  - 例: 1位記事タイトルが37文字なら、本記事は 32〜42 文字
- Google の検索結果表示上限（PC で約32〜35文字）でカットされても、続きの語句がCTRに有効なことがあるので、必ずしも32文字以下に収める必要はない
- 1位記事タイトル長が極端な場合（20文字未満や45文字超）は ±5 ではなく **28〜40 文字** の標準レンジに収める

#### 区切り記号（タイトル固有のルール）
- **`｜`（縦棒）は禁止**
- **`、`（読点）はタイトルには使わない**（H2では使うが、タイトルでは情報密度が下がるため避ける）
- **`【】`（角括弧）は積極的に使ってよい**: 年号・バリエーション・カテゴリ括りに有効
  - 例: `【2026年最新】` `【保存版】` `【完全版】`
- **`！`は積極的に使ってよい**: 言い切り・訴求のフックとして
- **`？`は使ってよい**: 疑問型タイトルで（1回まで）
- 半角スペース `　` (全角) や `・` も区切りとして使える

#### 必須要素
タイトルには以下を**できる限り入れる**（無理に全部入れず、必須要素の優先順位は1 → 4）:

1. **メインキーワードの構成語**（最優先・全ワード入れるのが基本）
   - 「AGA オンライン診療 おすすめ」なら **3ワード全部** をタイトルに含める
   - 2ワードしか入れないのは禁止。完全一致が文として不自然な場合のみ、語順を緩めて全ワードを含める
   - NG: `AGAオンラインおすすめ院を比較` （「診療」が抜けている）
   - OK: `AGAオンライン診療おすすめ17院！料金で選ぶランキング`（全3ワード含有）
   - **重要**: **キーワードはタイトルの左側に配置**する。Google SEO は左側のワードを重視する
     - OK: `AGAオンライン診療おすすめ17院！【2026年5月最新】料金で選ぶランキング`（KW左寄せ）
     - NG: `【2026年5月最新】AGAオンライン診療おすすめ17院！料金で選ぶランキング`（KWが角括弧の後ろ）
2. **具体数字**（院数・件数・ステップ数など）
   - 例: 「17院」「5つのポイント」「6つの判断基準」
   - 1位記事の数字（例: 17院）を踏襲するか、本記事の実数で書く
3. **年号と作成月**（任意・1位記事にあれば踏襲）
   - **必須ではない**。1位記事のタイトルに年号があれば本記事も入れる、なければ無理に入れない
   - 入れる場合の形式: `【{year}年{month}月最新】` のように **年号 + 作成月** をセットにする
     - OK: `【2026年5月最新】`
     - NG: `【2026年版】`（月がない）
     - NG: `【2026年最新】`（月がない）
   - 料金・ランキング・比較記事で鮮度が重要な場合のみ採用
   - Know系（仕組み・原因・効果）は年号不要
   - 必ず `{current_year}` 年を使う（過去年は禁止）
4. **コピーライティングフック**
   - 例: 「迷わず選ぶ」「後悔しない」「自分に合う」「失敗しない」「賢く選ぶ」
   - 1位タイトルにあるフレーズはそのままコピーせず、別の言い回しに変える
   - 頻出表現（「徹底比較」「完全網羅」「最強」「決定版」）は煽り感が強いので避ける

#### 1位記事タイトルとの関係（重要）

タイトル作成は **必ず1位記事タイトルを起点に**する:

1. 1位記事タイトルを読み、以下を抽出する:
   - 使われている年号（「2026年最新」など）
   - 数字フォーマット（「17院」「ランキング」など）
   - 訴求軸（「料金が安い」「失敗しない」「人気」など）
   - 区切り記号や記号（`！` `【】` など）
2. 抽出した要素を本記事タイトルに**取り入れる**
3. ただし表現は変える:
   - 同じ意味の別語彙を使う（「徹底比較」→「迷わず選ぶ」）
   - 語順を入れ替える
   - 1位タイトルの特徴的なキャッチをそのまま転用しない

#### 例: 1位記事 vs 本記事 タイトル対比（AGA オンライン診療 おすすめ）

| 観点 | 1位記事 | 本記事（推奨） |
|---|---|---|
| 原文 | `AGAオンライン診療おすすめランキング【2026年最新】17院で料金が安いのは？` | `AGAオンライン診療おすすめ17院！【2026年5月最新】料金と処方薬で迷わず選ぶランキング` |
| 文字数 | 40文字 | 45文字（±5） |
| キーワード位置 | タイトル左端 | タイトル左端（同じ） |
| キーワード | AGA / オンライン診療 / おすすめ 全含有 | AGA / オンライン診療 / おすすめ 全含有 |
| 数字 | 17院 | 17院（踏襲） |
| 年号 | 【2026年最新】（月なし） | 【2026年5月最新】（年号＋作成月） |
| 区切り | `【】` `？` | `【】` `！` |
| フック | 「料金が安いのは？」（疑問型） | 「迷わず選ぶ」（言い切り型・`！`で強調） |
| 訴求軸 | 料金 | 料金＋処方薬（広げる） |

### 3-5. クリニック/商材の掲載数（比較系のみ）
- トレース元の掲載数を踏襲（±1院まで）
- 競合で多く取り上げられているクリニックを優先

### 3-6. H3の表記ルール
- クリニック個別 H3 は **クリニック名のみ**。`クリニック名｜キャッチコピー` 形式は **禁止**
- OK: `<h3>AGAスキンクリニック 大阪梅田院</h3>`
- NG: `<h3>AGAスキンクリニック 大阪梅田院｜梅田駅徒歩1分の発毛実感率99.4%専門院</h3>`
- 「以下同様」「以下繰り返し」「テンプレートで15院を紹介」などの省略表現は **禁止**

### 3-7. FAQ セクションの設計
- Q&A は **最大5個まで**
- 本当に検索需要がある疑問に絞る
- 似たような質問の繰り返し・本文で既に答えている内容は除外

### 3-8. 重複・冗長セクションの禁止
- 内容が大きく被るセクションを2つ以上作らない
- 典型 NG（必ず統合する）:
  - 「結論」と「まとめ」を独立H2で両方置く → 1つに統合
  - 「料金一覧」と「費用比較表」を別H2にする → 1つのH2内にH3で展開
  - 「選び方」と「失敗しない選び方」「比較ポイント」を別H2にする → 1つに統合
  - 「メリット」と「特徴」「魅力」「強み」を別H2にする → 1つに統合
  - 「FAQ」と「よくある質問」を別H2にする → 1つに統合

### 3-9. 禁止事項（Step 2）
- H3の省略記法（「以下同様」「以下繰り返し」「テンプレートで○院紹介」）
- 根拠のない地域固定化（キーワードに地域名がないのに地域特化にする）
- 「要確認」前提の設計
- 中括弧のテンプレート記法・後で差し替える前提の仮置き文言を残す
- 仮置き記号（`◯ヶ月` `○選` `〇つ`）を見出しに残す

### 3-10. H2 見出しのコピーライティングルール

H2 は「SEO的に正しい」だけでなく、**読者が読みたくなる**コピーであることが重要。AI に任せると無難な見出しになりがちなので、以下のルールで明示する。

#### 区切り記号
- **`｜`（縦棒）は禁止**
  - NG: `AGAオンライン診療おすすめクリニック17院｜料金・処方薬で徹底比較`
- **基本は `、` か体言止めで自然に書く**。`！` `？` は **必要な時だけ意図的に**使う
  - 標準（記号控えめ）: `AGAオンライン診療おすすめ17院、料金と処方薬で迷わず選ぶランキング`
  - 標準（体言止め）: `AGAオンライン診療の選び方、後悔しないための6つの判断ポイント`
  - `！` `？` を使う場面: 強い訴求やフックを付けたいH2（記事内で2〜3個程度）
- **`！` `？` を全H2に機械的に付けない**。多用すると煽り感が出て信頼性が下がる
- 短いH2（10〜20文字程度）には記号を入れず、簡潔さを優先する

#### 文字数の目安
- **25〜40文字**（標準レンジ・本記事の主役H2）
- **短いH2(まとめ・FAQなど)は10〜25文字でも可**。短くて自然な方が良い場面では無理に伸ばさない
- 短すぎると情報不足、長すぎると判読しにくい

#### 表現パターン（3つから選ぶ・記号は必須ではない）

タグ構成設計時に、各 H2 ごとに以下のどれを採用するかを判断する。`！` `？` を使うかは型と関係なく、**意図的に必要な時だけ**選ぶ。

**A. 言い切り・オススメ訴求型**
- 結論や推奨をストレートに提示
- 標準形（記号なし）: `AGAオンライン診療おすすめ17院、料金と処方薬で迷わず選ぶランキング`
- 強訴求形（`！`使用）: `AGAオンライン診療は17院を比べれば迷わない！料金と対応で選ぶランキング`
- 用途: ランキング・おすすめH2、料金比較H2、結論系H2

**B. 読者の疑問に寄り添う型**
- 読者の不安・疑問を見出しで代弁
- 標準形（記号なし）: `AGAは対面とオンライン、結局どちらが良いのか5つの観点で比較`
- 疑問符形（`？`使用）: `AGAは対面とオンライン、結局どっちが良い？5つの観点で違いを比較`
- 用途: 比較H2、流れ・手順H2

**C. ターゲッティング型**
- ターゲット読者を見出しで明確化
- 例: `会社にバレずに始めたい人へ、おすすめのAGAオンライン診療3院`
- 例: `初めてのAGAオンライン診療、失敗しない選び方6つのポイント`
- 用途: ペルソナ別の選び方H2、ニーズ特化H2

#### H2用途別の推奨スタイル

| H2の役割 | 推奨スタイル | 例 |
|---|---|---|
| メインランキング・おすすめ | A 言い切り型（`！`を意図的に使うのは可） | `AGAオンライン診療おすすめ17院、料金と処方薬で迷わず選ぶランキング` |
| 選定基準・選び方 | A or C | `AGAオンライン診療の選び方、後悔しないための6つの判断ポイント` |
| 料金比較 | A 言い切り型 | `料金で選ぶAGAオンライン診療、安いクリニックの賢い選び方` |
| 比較・違い | B 疑問型 | `AGAは対面とオンライン、結局どっちが良い？5つの観点で違いを比較` |
| 流れ・手順 | C ターゲッティング型 | `AGAオンライン診療、申し込みから薬の到着までの流れを5ステップで解説` |
| **FAQ** | **短いA言い切り型（記号なし推奨・10〜20文字）** | `AGAオンライン診療でよくある質問` |
| **まとめ** | **短いA言い切り型（記号なし推奨・15〜25文字）** | `まとめ、AGAオンライン診療は3つの軸で選ぶ` |

FAQ・まとめは **記事の補助的役割** なので、メインH2のような訴求コピーは不要。短く・キーワードを自然に含むだけで十分。

#### キーワード含有ルール
- メインキーワードの構成語（例: `AGA` `オンライン診療` `おすすめ`）は **なるべく全H2に入れる**
- 自然な日本語で入れられない場合は例外を許容するが、その判断は最小限に
- まとめH2にも入れる（例: NG `まとめ` / OK `まとめ、AGAオンライン診療は料金とサポートで選ぶ`）

#### 禁止表現
- 1位記事の見出しを丸ごとコピー
- 1位記事の固有キャッチコピー（「クーポンで6ヶ月実質無料！」「業界最安962円」など）を**そのまま**転用 → 同じ事実を別表現で書く
- 仮置き記号（`◯` `○`）を残す
- 「徹底比較」「完全網羅」「最強」など**手垢のついた表現の機械的多用**（1記事内で2回以上使わない）

#### タグ構成設計書での記載方法

各 H2 に「型」を明記する:

```markdown
### [H2] AGAオンライン診療おすすめ17院！料金と対応で迷わず選ぶランキング
※トレース根拠: 1位記事の「AGAオンライン診療おすすめランキング【2026年版】」に対応
**コピー型**: A. 言い切り型
**目標文字数**: ...
```

これで、別のチャットで再設計するときも同じパターンで書ける。

## 4. Step 3 — 本文HTML生成のルール

### 4-1. 出力ルール
- HTMLのみ出力（マークダウン・説明文・コードフェンス禁止）
- CSSクラス名だけ付与する。**本文HTML内に `<style>` `<script>` タグを書かない**
  - **例外**: なんでもサイト等の **早見表ショートコードHTML** は `<style>` を内包したまま出力する。これは「ショートコードの完成済みHTML」を本文に貼り付ける運用のため
  - 早見表ショートコード以外で `<style>` を書くのは禁止（クリニックH3・テーブル・口コミブロックなどの本文構造には絶対に書かない）
- ショートコードはタグ構成設計書で必要なものだけ出力
- 中括弧のテンプレート記法・後で差し替える前提の仮置き文言を一切出力しない
- 参考文献一覧・参考リンクはこのステップで入れない（後段 `add_references.py` で付与）
- 不明な情報には `※要確認` や説明コメントを出力しない。確認できない項目は **その文・行・セルを省略する**
- プレースホルダー文字列を残さない:
  - **NG**: `料金: 来院前に確認` / `診察料: 公式サイトで確認` / `薬剤ごとに確認` / `通いやすい時間帯を確認しやすい`
  - **対応**: 公式情報で埋められない行はテーブルから削除、または該当院を掲載から外す

### 4-2. 導入部分（H2の前）
1. リード文（2段落）
   - 1段落目: 検索意図への共感
   - 2段落目: この記事で何がわかるかの提示
2. タグ構成設計書の導入構成にショートコードがある場合のみ出力
3. タグ構成設計書の導入構成に一覧ボックスがある場合のみ、`clinic-index-box` の実HTMLを出力（`.clinic-list-box` 等の独自クラスは作らない）
4. `<!-- ↑ここまでが導入部分。この直後にWordPress自動生成の目次が入る -->`
5. **本文先頭に `*_top.png` を直接 `<img>` で入れない**（WPアイキャッチでのみ使用）

### 4-3. H2 セクション
- タグ構成設計書で定義された H2 をそのまま使う
- **タグ構成設計書にある H2 を1つも省略しない**（厳守）
- タグ構成設計書にない H2 を勝手に追加しない
- H2 見出しの直下に `<img src="images/...">` を入れない（自動画像生成は廃止済み）

### 4-4. クリニック/商材 個別セクション（H3）のテンプレート

タグ構成設計書が比較記事・おすすめ記事で、クリニック/商材の個別 H3 セクションを定義している場合のみ、以下を組み合わせて出力する。**何を入れるかはタグ構成設計書のブロック構成と1位記事の構造に従う**（盲目的に全部出さない）。

<!-- ============================== -->

<!-- 公式サイトのスクリーンショット（タグ構成設計書が指定する場合のみ） -->

**ルール:**
- **summary_table と detail_table の両方を必ず両方出す必要はない**。タグ構成設計書のメモに従う（1個でも OK）。ただし **両方0個は禁止**
- クリニック個別 H3 内に `treatment-compare-table` を入れない（H2直下の横並び比較でのみ使う）
- オンライン専門クリニック（店舗なし）は Google マップを挿入しない。`<!-- クリニック名はオンライン専門のためマップなし -->` のコメントを入れる
- 公式サイトURLが不明な場合は、CTAボタンそのものを出力しない
- 「以下同様」「以下繰り返し」「テンプレートで○院紹介」などの省略表現は禁止

### 4-5. テーブルの CSS クラスルール

| 配置場所 | クラス名 | 構造 |
|---|---|---|
| H3 セクション内の最初のテーブル（院プロフィール） | `clinic-summary-table` | `<tbody>` のみ、th\|td 2列 |
| H3 セクション内の2番目のテーブル（院詳細） | `clinic-detail-table` | `<tbody>` のみ、th\|td 2列 |
| H2 直下のクリニック比較テーブル（「○選」記事の主軸表） | `clinic-list-tbl` | `<thead>` 付き、上位5院、ロゴ＋公式ボタン |
| H2 直下のその他比較テーブル（治療法比較等） | `treatment-compare-table` | `<thead>` 付き |

- 比較テーブルは `<div style="overflow-x:auto;">` で囲む
- テーブルに inline `<style>` を入れない（CSS は WP の「カスタムCSS & JS」欄に配置）

### 4-6. `clinic-list-tbl` の必須構成（H2直下のクリニック比較表）

「○○のクリニック○選」「クリニック比較」系記事の主軸となるH2直下の比較表は、必ずこの形式:

- **掲載数: 上位5院のみ**（記事内に11院いる場合でも、比較表は最も推薦する5院だけに絞る）
- **1列目: クリニック名＋ロゴ＋公式ボタン**を `<div class="cl-cell">` で縦並び
- **ロゴ画像**: `images/logo_{クリニック識別子}.png` を `<div class="cl-logo-box">` 内に表示
  - ロゴ素材は `capture_logos.py`（pipeline.py Step 5.5）が自動取得・全記事キャッシュ共有
  - Step 3 時点ではロゴ画像未取得なので、`<div class="cl-logo-text">クリニック名</div>` のテキスト代替で出力。後段で `<img>` に差し替えられる
- **公式ボタン**: `<a class="cl-btn">公式</a> rel="noopener noreferrer sponsored"` 必須
- **価格セル**: `<span class="price-em">` で赤強調、補足は `<span class="sub">` で小さく表示
- **クリニック名**: `<a href="#clinic-xxx">` で記事内詳細にアンカー
- **上位5院の選定**: 記事内H3順序の先頭5院（最も推薦する5院）

`clinic-list-tbl` の HTML テンプレートは Step 3 で具体ファイルとして注入する。CSS は WP 側に配置。

### 4-9. テキストのライティングルール

#### 段落の長さ
- 1パラグラフ **80〜100文字以内**
- 長くなる場合は `<strong>` で視覚的にカバー

#### 文体
- 事実ベースで書く。「〜と言われています」「〜のようです」等の曖昧表現を避ける
- 各セクションの冒頭1文で結論を述べ、その後に補足する構成
- 「です・ます」調で統一
- **客観整理メタ表現は禁止**（§1-8参照）

#### `<strong>` 強調ルール
**50文字前後に1箇所**の頻度で `<strong>` を使う。短い段落でも最低1箇所は入れる。

強調すべきポイント:
- 価格情報（「初月○○円」「月々○○円」等）
- 差別化ポイント（「予約不要」「完全個室」「22時まで」等）
- 数値を伴う実績（「99.4%」「250万人」等）
- 制度・サービス名（「全額返金保証」「主治医制度」等）

注意:
- 文章の内容は変えない。視覚的なメリハリのみ追加
- `<strong>` には WP 側 CSS で蛍光マーカー風アンダーラインが適用される

#### 番号体系の整合
- 同じ H2/H3 セクション内で「`1. ○○`」のような番号付き太字見出しと、別の番号付きリスト `<ol>` を**同時に並べない**
- 並ぶと番号体系が衝突して読み手が混乱する
- 推奨パターン: 「リスト全体を `<ol>` で先に提示 → そのうち1つを個別詳述」の順

### 4-10. 詳細テーブル内の表記ルール

`clinic-detail-table` 内で複数の薬や料金を列挙する場合、読点（、）ではなく `<br>` 改行:

```
変換前: フィナステリド30錠：3,960円、デュタステリド30錠：6,930円
変換後: フィナステリド30錠：3,960円<br>デュタステリド30錠：6,930円
```

### 4-11. 比較表の横長セルは `<br>` で改行

`treatment-compare-table` などの比較表で長文セル（料金プランの列挙・複数院の住所併記など）は適度な区切りで `<br>` 改行を入れる。

区切りの目安: プラン名のあと、「他…」のあと、円表記の区切り、治療法・条件の切り替わり点

### 4-12. クーポン適用後の特価を主表示しない

- 比較表・料金プラン表で「クーポンコード適用後の特価」を主表示にしない
- 通常価格を主、クーポン適用後の値段は本文や脚注で補足扱い
- 早見表ボックス（`ndm-aga-v*` テンプレ等）の冒頭・末尾の比較表でも同様
- クーポン依存の数値は、提携先のクーポンが切れると虚偽表記になるため避ける

### 4-13. ショートコード

ジャンル設定の `shortcodes` を参照:
- 「早見表」: 渡された値が完成済みHTMLならそのまま出力、ショートコード文字列の場合は `[sc name="..."][/sc]` をそのまま出力
- 「なんでも」サイトの早見表は **完成済みHTMLを本文内に直接出力**（ショートコード文字列を残さない）
- 「一覧ボックス」はプレースホルダーではなく実際のHTML
- タグ構成設計書で不要なショートコードは無理に出力しない

### 4-14. まとめセクション

記事末尾に以下を配置:
1. まとめテキスト（2段落）
2. タグ構成設計書で CTA やショートコードが指定されている場合のみ出力
3. 注意書き: `<p><small>※本記事に記載の料金はすべて税込表記です。料金・診療時間・治療内容等は変更される場合がありますので、最新情報は各クリニックの公式サイトをご確認ください。</small></p>`

### 4-15. 末尾 CTA フォーマット（fill_final_cta.py 出力）

```html
<!-- final-cta:start -->
<div class="article-final-cta__title">{見出し}</div>
<p>{lead_1}</p>
<p>{lead_2}</p>
{早見表ショートコード（または resolve 後のHTML）}
<p><small>※注意書き</small></p>
<!-- final-cta:end -->
```

**禁止:**
- `<p class="article-final-cta__eyebrow">COMPARE FIRST</p>` などの eyebrow テキスト
- 外側の `<div class="article-final-cta cta-variant-1">...</div>` ラップ（背景グラデーション・ボックス装飾）

タイトルは `<h2>` ではなく `<div class="article-final-cta__title">` を使う（WP自動目次に出さないため）。

## 5. ジャンル設定ファイル（`genres/{genre_id}.json`）

ジャンルごとに以下を定義:

| フィールド | 説明 | 例（AGA） |
|---|---|---|
| `genre_id` | ジャンル識別子 | `aga` |
| `genre_name` | ジャンル表示名 | `AGA治療` |
| `default_article_type` | デフォルト記事タイプ | `クリニック比較` |
| `supported_article_types` | 対応する記事タイプ | `["クリニック比較", "解説記事", "費用解説記事", "対処法記事"]` |
| `shortcodes` | 早見表などのショートコード | `{ "早見表": "[sc name=\"aga-hayamihyou\" ][/sc]" }` |
| `summary_table.columns` | 概要テーブルのカラム名 | `["予防プラン", "発毛プラン", "診療時間", "アクセス", "オンライン診療"]` |
| `detail_table.columns` | 詳細テーブルのカラム名 | `["クリニック名", "AGA治療薬", "診察料", "支払い方法", "住所"]` |
| `text_requirements` | テキストに含めるべき情報（候補リスト、必須ではない） | `["強み", "料金メリット", "立地", "治療法", "向いている人"]` |
| `factcheck_fields` | ファクトチェック対象項目 | `["価格", "住所", "診療時間", "アクセス", "診察料", "支払い方法"]` |
| `notes` | 記事末尾の注意書き | `税込表記・公式サイト確認の旨` |

**重要:** `text_requirements` は **候補リスト**であり、すべて埋める必須リストではない。各クリニック紹介で何を書くかは、**1位記事の同クリニック紹介で扱われている項目**を優先候補にする。競合が触れていない項目を text_requirements にあるからという理由だけで盛り込まない（水増しになる）。

新規ジャンル追加時: `genres/` に JSON ファイルを追加するだけで対応可能。

## 6. バリアント記事の規約（v2〜v5）

`__nandemo_v2`〜`v5` のような同一キーワードの別バリアント記事を生成するときは、**必ず title タグとメタディスクリプションを v1 と別物にする**。本文だけ書き換えて title/meta が共通だと、検索エンジンに重複コンテンツとして認識されてしまう。

### 必須事項
1. `output/{slug}__nandemo_v{N}/{slug}__nandemo_v{N}_タグ構成.md` を必ず作成
2. その中の `titleタグ` と `メタディスクリプション` を v1 と**異なる訴求軸・別語順**で書く
3. リード文も差別化する（v1 のリードコピーを流用しない）
4. `build_for_wp.py` は同フォルダの `*_タグ構成.md` から title/meta を抽出し、`_記事_for-wp.html` の冒頭に HTMLコメントとして差し込む

### 訴求軸の振り分け例
| バリアント | 訴求軸 |
|---|---|
| v1 | 初月料金の安さ・全額返金保証 |
| v2 | 12ヶ月総額・続けやすさ |
| v3 | 対面 vs オンラインの選び分け |
| v4 | 治療法の幅・進行度別の選び方 |
| v5 | 受診のしやすさ・夜間/土日対応 |

データテーブル・画像・テーブルレイアウトはバリアント間で使い回してよい。差別化するのは title／meta／リード／クリニック紹介の地の文／まとめ／FAQ回答などの**散文**部分。

### バリアント別の見た目テーマ

v1〜v5 は本文の散文だけでなく**ビジュアルも別物**にする。`output/article-theme-v{N}.css` を `build_for_wp.py` が自動で base CSS の後ろに連結注入する。

| variant | テーマ | 主色 | 蛍光マーカー |
|---|---|---|---|
| v1 | Cool Navy | `#1e3a8a` ネイビー | 黄 `#fef08a` |
| v2 | Warm Orange | `#c2410c` 燃え赤 | ピーチ `#fed7aa` |
| v3 | Earthy Beige | `#78350f` ブラウン | クリーム `#fef3c7` |
| v4 | Soft Editorial Blue | `#89a6c9` ダスティブルー | アプリコット |
| v5 | Modern Lavender | `#75668f` ラベンダー | ラベンダー `#e8dff4` |

**v1 はテーマファイル不要**: `article-common.css` がそのまま v1 の Cool Navy テーマになっているため、`article-theme-v1.css` は作らない。

### AGA 早見表のバリアント運用

`なんでも` の AGA 記事では、本文内の比較早見表だけでなく導入部・末尾 CTA に入る早見表も variant ごとの HTML を source of truth とする。

- `v1` は `shortcodes/nandemo/aga-hayamihyou-1.html`
- `v2` は `shortcodes/nandemo/aga-hayamihyou-2.html`
- `v3` は `shortcodes/nandemo/aga-hayamihyou-3.html`
- `v4` は `shortcodes/nandemo/aga-hayamihyou-4.html`
- `v5` は `shortcodes/nandemo/aga-hayamihyou-5.html`

### V3 完成ルール

V3 は現行デザインを正式版として固定する。今後 V3 を再生成・複製するときは、生成済み HTML を都度手修正せず、必ず下記の source of truth を編集して反映する。

- 本文テーマの正本: `output/article-theme-v3.css`
- AGA 早見表の正本: `shortcodes/nandemo/aga-hayamihyou-3.html`
- `*_記事_for-wp.html` は配布物であり、V3 の恒久ルール定義ファイルとしては扱わない

V3 の固定仕様:
- `.ndm-article` の背景は `transparent` を維持（本文全体にうっすらしたベージュ背景を敷かない）
- スマホ時の文字サイズは V3/V4/V5 で揃える（本文 14px / h2 18px / h3 17px）
- V3 の違いは主に行間と配色・装飾で表現

### V4 / V5 末尾 CTA ルール

V4 と V5 の末尾 CTA は、外側に `.article-final-cta` ラッパーを持たせず、本文の流れにそのまま接続するプレーン配置を正とする。

- V4: 説明2段落 + `aga-hayamihyou-4.html` + 注意書き
- V5: 説明2段落 + `aga-hayamihyou-5.html` + 注意書き

## 8-2. H2画像振り分け（後工程：H2配下に画像を自動挿入してWP下書きを更新）

記事は先に本文だけ生成し、ChatGPT などで作った H2 用ビジュアル画像を後から差し込むケースが多い。そのときの **標準フロー** をここに固定する。チャットやAIごとに進め方がぶれないよう、必ず以下に従う。

### 8-2-1. いつ使うか

- 記事HTML（`output/<slug>/<slug>_記事.html`）が完成している
- ChatGPT などで作った H2 用画像（各H2のタイトルと連動したコピー入りビジュアル）が手元にある
- それらを WP メディアにアップロードして、対応するH2の直下に差し込みたい

### 8-2-2. 入力

| 項目 | 値 |
| --- | --- |
| 記事 | `output/<slug>/<slug>_記事.html`（_for-wp.html 前の元ファイル） |
| サイト設定 | `sites/<site>.json` |
| 画像 | ローカルのファイルパスを **明示指定**（通常は `~/Downloads/` 配下） |
| WP下書き | 原則「記事タイトルで検索」で自動特定。複数ヒットしたら明示指定 |

ローカルの「ダウンロードフォルダの直近N分」や「ディレクトリまるごと」は使わない。**関係ない画像が混ざるリスクが高いため、必ずファイルを明示指定する**。

### 8-2-3. 標準コマンド

```
python3 scripts/insert_h2_images.py \
  --article "output/<slug>/<slug>_記事.html" \
  --site sites/<site>.json \
  --images ~/Downloads/img1.png ~/Downloads/img2.png ...
```

オプション:

- `--wp-post-id <ID>`: WP投稿IDを明示指定（タイトル検索で複数ヒットしたとき）
- `--dry-run`: WP更新せず、画像コピー抽出・マッチング・HTML書き換えまでで止める
- `--skip-rebuild`: `build_for_wp.py` を呼ばない（既に `_for-wp.html` がある場合）

### 8-2-4. 内部処理（このスクリプトが必ず守る順序）

1. **画像コピー抽出** — Claude Vision で各画像内のテキスト（タイトル・コピー・ラベル）を読み取る
2. **WPメディアアップロード** — `.image_upload_cache.json` に SHA-256 をキーとして保存し、**同じ画像は再アップしない**
3. **H2抽出** — 記事HTMLから全H2を出現順にリストアップ
4. **マッチング** — 画像コピー × H2タイトルを Claude に渡し、内容が最も近いペアだけ採用（曖昧なものは除外）。承認ステップは挟まず全自動
5. **HTML書き換え** — 各H2直下に `<figure class="wp-block-image size-large"><img src="..." alt="<H2タイトル>" class="wp-image-<ID>" /></figure>` を挿入。**後ろから挿入**してインデックスずれを防ぐ
6. **`_for-wp.html` 再生成** — `build_for_wp.py --html <記事>` を呼ぶ
7. **WP下書き更新** — タイトル検索で投稿を特定し、`wp_post.WordPressClient.update_post()` で更新。REST が500系で落ちても XML-RPC に自動フォールバック

### 8-2-5. 画像の規約

- **コピー入りビジュアル限定**: 画像内のテキストでH2と紐づける仕組みなので、文字のない汎用イラストには使えない
- **1H2につき1画像**: 同じH2に複数画像は入れない（マッチング側で除外）
- **アイキャッチや余り画像は差し込まない**: マッチングで該当H2がなければスキップされるので、不要な画像も `--images` に混ぜてOK（ただし無駄なアップは発生する）
- **altテキスト** = 当該H2の plain text（記号も含めそのまま）

### 8-2-6. 既知の挙動

- なんでもサイトは REST API `posts/<id>` が大きな本文で 500 を返すことがある（プラグイン由来）。`wp_post.py` の XML-RPC フォールバックで自動的に救済される。手で `requests.post()` を直接叩かない
- WP投稿のステータスはそのまま維持される（`draft` のままなら `draft` のまま）
- 記事HTML（`_記事.html`）は **インプレース書き換え**。再実行したい場合は git で戻すかバックアップを取ること
